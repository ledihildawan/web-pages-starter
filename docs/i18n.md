# i18n

The internationalization system in this project is built around a single source of truth for both copy and behavior. Source translations live as JSON under `locales`, while the full i18n runtime (locale data, formatters, i18next init, Alpine store) lives in `packages/i18n/` — a standalone, code-split package at the project root organized into `data/`, `engine/`, `strategies/`, `fonts/`, `runtime/`, and `config/` subdirectories. The Nunjucks SSR template lives in `packages/template-engine/` (one-way dependencies: `@template-engine` → `@i18n` and `@template-engine` → `@page-system`), keeping `packages/i18n/` a pure engine. All external imports use the `@i18n` alias (e.g. `@i18n/config/define`, `@i18n/engine/formatters`). The same keys are consumed by the Nunjucks template helpers for server-side rendering and by an i18next runtime that updates the DOM in place when the user switches language.

This document is the canonical reference for the system. It is accurate as of the file's "Last updated" line at the bottom and tracks the actual implementation, not aspirational behavior.

## Bundle Optimization

The `packages/i18n/` package is structured to keep the client bundle as small as possible. Multiple mechanisms work together:

### Build-time data filtering

`packages/i18n/cli/generate-active-locales.ts` generates `generated/active-locales-data.ts` from the master data in `packages/i18n/data/`:

- The generated file contains only the locale entries listed in `i18nConfig.locales` (plus the `defaultLocale`). Unused locales are stripped entirely.
- No dev stub — always uses `i18nConfig.locales` for dev, preview, and build.

All i18n data arrays (`LOCALES`, `LANGUAGES`, `REGIONS`, `CURRENCIES`, `TIMEZONES`, `WRITING_SYSTEMS`, etc.) flow through this generated file, so the full master tables are eliminated from the client bundle in production. Only the active subset ships.

### Single-locale optimization

When only 1 locale is active (`i18nConfig.locales` is empty or has 0 entries):
- Bootstrap skips i18next initialization entirely
- Alpine i18n store registered as stub (`{ current, languages: [], change: ()=>{} }`)
- No locale JSON fetch, no LanguageDetector, no i18next bundle
- Minimal client i18n script: `window.__SERVER_LOCALE__="en-US"`
- Navbar language switcher hidden via `{% if locales.length > 1 %}` template guard

### Generic font subset filtering

`scripts/generate-fonts-css.ts` generates `public/assets/fonts/fonts.css` with only the font subsets needed by active locales:

- Inter subsets: Latin → `[latin, latin-ext]`, Cyrillic → `[cyrillic, cyrillic-ext, latin, latin-ext]`, Greek → `[greek, greek-ext, latin, latin-ext]`
- Vietnamese extra subset: added when a Vietnamese locale (`vi`) is active
- Noto Sans per-script: each writing system gets its font + Latin fallback
- CJK fonts: all subsets included (character set too large to filter)
- `scripts/subset-fonts.ts` further reduces woff2 size via `pyftsubset` (removes hinting, unused OpenType tables)

### Code-split cardinal/ordinal strategies

Per-language cardinal and ordinal spellout (e.g. Arabic, Japanese, Chinese) lives in `packages/i18n/strategies/` (`builder.ts`, `loader.ts`, and per-language files `id.ts`, `ja.ts`, `zh.ts`, `ar.ts`). Each language strategy is its own async chunk, loaded on demand via dynamic `import()` only when a locale that needs it is active. Languages without a manual strategy fall back to `Intl.PluralRules` + the English suffix map, which ships in the core bundle.

### `@i18n` alias

All external imports use the `@i18n` alias so callers never reference the package's internal folder layout directly:

```ts
import { defineI18n } from '@i18n/config/define';
import { formatCardinal } from '@i18n/engine/formatters';
import { getActiveLocaleCodes } from '@i18n/engine/active-locales';
```

## Source Translations

### File layout

```txt
locales/
  en-US/                       # default locale
    common.json               # site-wide copy
    home.json                 # page-specific copy
    pricing.json
    cta.json                  # shared locale copy (auto-detected by scanSharedLocales())
  id-ID/
    common.json
    home.json
    ...
```

Locale files use `.json`. Data files (`data/`, `pages/*/index.json5`) use `.json5`. The `readJSON5` utility parses `.json` files as well, but the template and runtime paths always reference `.json` — there is no automatic `.json5` fallback for locale files.

### Namespace resolution

The template key prefix determines the i18next namespace at runtime.

| Template key | Runtime namespace | Source file |
| --- | --- | --- |
| `site_name` | `common:site_name` | `locales/{locale}/common.json` |
| `home:hero.title` | `home:hero.title` | `locales/{locale}/home.json` |
| `cta:heading` | `cta:heading` | `locales/{locale}/cta.json` |

The `i18next` default namespace is `common`. The page namespace and any shared locales are registered on `i18next.init`. Shared locales are auto-detected by `scanSharedLocales()` which scans templates for `i18n.t('namespace:...')` usage.

> **`page_id` is decoupled from the folder name.** The `page_id` (from `index.json5`) is the stable identifier used as the namespace and locale file key (`home.json`, `not-found.json`, etc.). System page folders use locale-dependent slugs from `SYSTEM_PAGE_SLUGS` — when the default locale changes, `sync-system-pages` renames the folders but the locale files stay keyed by `page_id` and are never renamed.

### Plural keys

i18next pluralization is keyed by suffix. The suffix for a given count is computed by `getPluralSuffix(count, locale?)` using the browser's `Intl.PluralRules` (no per-locale config required — categories are inferred from the active locale automatically):

| `Intl.PluralRules` category | i18next key suffix |
| --- | --- |
| `zero` | `_zero` |
| `one` | `_one` |
| `two` | `_two` |
| `few` | `_few` |
| `many` | `_many` |
| `other` | `_other` |

```json5
// locales/id-ID/home.json
{
  hero: {
    title: "Build multilingual pages faster",
    subtitle: "One translation source for server rendering and runtime switching.",
  },
  stats: {
    projects_one: "{{count}} proyek",
    projects_other: "{{count}} proyek",
    projects_zero: "0 proyek",
    projects_two: "{{count}} proyek",
    projects_few: "{{count}} proyek",
    projects_many: "{{count}} proyek",
  },
}
```

```njk
{{ i18n.t('home:hero.title') }}
{{ i18n.plural('home:stats.projects', 7) }}
```

### Variables

Variables use the `{{name}}` syntax and are interpolated by both the SSR `resolve()` and i18next at runtime.

```json5
{ welcome: "Hello {{name}}" }
```

```njk
{{ i18n.t('welcome', { name: 'Dinda' }) }}
```

### System pages

System pages are the 7 pages whose folder names are locale-dependent: `home` (root), `not-found`, `unauthorized`, `forbidden`, `server-error`, `maintenance`, `offline`. They are defined in `packages/page-system/system-pages.ts`:

| Export | Purpose |
| --- | --- |
| `ROOT_PAGE` | Stable page_id of the root page (always `'home'`) |
| `SYSTEM_PAGE_IDS` | List of all 7 system page ids |
| `SYSTEM_PAGE_SLUGS` | Per-page, per-locale URL slugs (7 pages x 136 locales) |
| `getSystemPageSlug(pageId, locale)` | Resolve a system page's slug for a given locale |
| `getRootPageSlug(locale)` | Resolve the root page's slug for a given locale |
| `getErrorPageSlugs(locale)` | Array of error page slugs (excludes root) for a given locale |
| `isSystemPageId(pageId)` | Check if a page_id is a system page |
| `isSystemPageSlug(slug, locale)` | Check if a slug matches a system page for a given locale |

The `page_id` (from `index.json5`) is the stable identifier — it drives the i18n namespace and locale file key. Locale files (`home.json`, `not-found.json`, etc.) are keyed by `page_id` and are never renamed. The folder name is the locale-dependent slug derived from `SYSTEM_PAGE_SLUGS`.

When the default locale changes, `packages/page-system/cli/sync-system-pages.ts` renames ALL system page folders to the new locale's slugs. Non-Latin slugs are romanized to URL-safe ASCII via `scripts/lib/romanize.ts` (limax-based). For example, with `defaultLocale: 'id-ID'`, the root page folder becomes `beranda` (Indonesian for "home"), but locale files remain `home.json` across all 136 locales.

The service worker (`public/sw.js`, generated by `scripts/generate-sw.ts`) embeds locale-specific error page URLs from `getErrorPageSlugs()` so the precache list matches the default locale's folder names.

## Template API

The Nunjucks `i18n` object is created in `packages/template-engine/template.ts` and is available to every template and partial that extends `main.njk`.

### Translation API

| API | Returns | Notes |
| --- | --- | --- |
| `i18n.t(key, vars?, options?)` | Wrapped HTML | `data-i18n` span; refreshes on language change. Pass `{ raw: true }` for plain string |
| `i18n.html(key, vars?)` | Raw locale string | Returns the resolved string with no wrapper (useful inside attribute values) |
| `i18n.attr(key, attrName, vars?)` | `attrName="…"` | Emits a complete attribute, e.g. `alt="…"` |
| `i18n.plural(key, count, vars?, options?)` | Wrapped HTML | `data-i18n-plural` + `data-i18n-count`; refreshes on language change |
| `i18n.getPluralSuffix(count)` | `_one` \| `_other` \| `_few` \| `_many` \| `_zero` \| `_two` | Useful for direct key construction |

`i18n.t()` options:

| Option | Effect |
| --- | --- |
| `raw: true` | Return a string without the wrapper span |
| `native: true` | Force native digits in the output (`data-force-native`) |
| `universal: true` | Force Latin digits (`0-9`) in the output (`data-force-universal`) |
| `className: '…'` | Add a class to the wrapper span |

`i18n.plural()` options: `raw`, `className`. The `count` is automatically added to the `vars` and the correct suffix is appended to the key for SSR.

Example output:

```html
<span data-i18n="home:hero.title">Build multilingual pages faster</span>
```

### Formatting API

Every formatter renders a `<span>` with one or more `data-format-*` attributes. `updateFormattedElements()` walks the DOM and replaces the inner text whenever the active locale changes. Pass `{ raw: true }` to receive a plain string.

| API | Data attribute | Notes |
| --- | --- | --- |
| `formatNumber(value, options?)` | `data-format-number` | `Intl.NumberFormat` |
| `formatCurrency(value, currency, options?)` | `data-format-currency` + `data-target-currency` | Currency code is required |
| `formatPercent(value, options?)` | `data-format-percent` | `0.85` → `85%` |
| `formatBytes(bytes, decimals = 1, options?)` | `data-format-bytes` + `data-bytes-decimals` | Uses binary units (`KiB`-style but labeled `KB`) |
| `formatDuration(seconds, options?)` | `data-format-duration` | Combines `Intl.RelativeTimeFormat` and `Intl.ListFormat` |
| `formatDate(date, options?)` | `data-format-date` | Uses locale timezone and calendar |
| `formatTime(date, preset?, options?)` | `data-format-time` + `data-time-preset` | `preset` is `'short'` \| `'medium'` \| `'long'` \| `'full'` |
| `formatDateTime(date, options?)` | `data-format-datetime` | Combines date and time |
| `formatOrdinal(value, options?)` | `data-format-ordinal` | Manual strategies for `id`, `ja`, `ar`; `Intl.PluralRules` + English suffix map for all other locales |
| `formatCardinal(value, options?)` | `data-format-cardinal` | Manual strategies for `id`, `ja`, `zh`, `ar`; otherwise returns the raw number string (`Intl` has no API for cardinal spellout). `options.gender: 'masculine' \| 'feminine'` only affects the Arabic strategy. |
| `formatScientific(value, options?)` | `data-format-scientific` | `Intl` scientific notation with native digit fallbacks |
| `formatAbbreviated(value, options?)` | `data-format-abbreviated` | Compact form such as `15.3K` |
| `formatList(items, options?)` | `data-format-list` + `data-items` | `Intl.ListFormat` |
| `formatUnit(value, unit, options?)` | `data-format-unit` + `data-unit` | `Intl.NumberFormat` with `style: 'unit'` |
| `formatRelativeTime(value, options)` | `data-relative-time` + `data-unit` + `data-numeric` | `options.unit` is required; `options.numeric` is `'always'` \| `'auto'` (default `'always'`) |
| `nativeDigits(text, force?, options?)` | `data-native-digits` + `data-force` | Initial conversion + refreshed at runtime |
| `pluralize(word, count?, inclusive?, options?)` | `data-pluralized` + `data-count` + `data-inclusive` | Initial English pluralization; refreshed at runtime |
| `singularize(word, options?)` | `data-singularized` | Initial English singularization; refreshed at runtime |
| `getPluralSuffix(count)` | – | Returns the suffix string |

Common options supported across the numeric formatters:

| Option | Effect |
| --- | --- |
| `raw: true` | Return a string without a wrapper span |
| `className: '…'` | Add a class to the wrapper span |
| `nativeDigits: true` | Render in the locale native numbering system |
| `numberingSystem: 'latn' \| 'arab' \| 'deva' \| …` | Override the `Intl` numbering system |
| `style: 'currency' \| 'percent' \| 'unit' \| …` | Standard `Intl` options |

Any `Intl.NumberFormatOptions` or `Intl.DateTimeFormatOptions` key is forwarded to `Intl.*Format` so things like `maximumFractionDigits`, `dateStyle: 'long'`, and `timeZoneName: 'short'` work without explicit support here.

```njk
{{ i18n.formatDate('2026-06-04', { dateStyle: 'long' }) }}
{{ i18n.formatNumber(1200, { maximumFractionDigits: 0 }) }}
{{ i18n.formatTime(page.published_at, 'medium') }}
```

### Grammatical gender (Arabic cardinals)

Arabic cardinal numerals change form based on the gender of the noun they modify. The `formatCardinal()` formatter accepts an optional `gender` field for Arabic locales; it is ignored for `id`, `ja`, `zh`, and the `Intl` fallback. Default is `'masculine'`.

```njk
{{ i18n.formatCardinal(2) }}
{# → "اِثنان" (masculine, default) #}

{{ i18n.formatCardinal(2, { gender: 'feminine' }) }}
{# → "اِثنتان" #}

{{ i18n.formatCardinal(99, { gender: 'feminine' }) }}
{# → "تِسع وتِسْعون" #}
```

The Arabic strategy covers `0`–`999,999,999,999` (units, teens, tens, hundreds, thousands, millions). Larger numbers fall back to the raw digit string. To extend the strategy (or add support for `Intl.PluralRules` integration with `formatCurrency`), edit `packages/i18n/engine/formatters.ts` (or add a code-split strategy under `packages/i18n/strategies/`).

### Pricing API

Pricing plans use a regional pricing object. `pricing.base` is the USD fallback. Add a per-locale entry when local pricing exists.

```json5
{
  pricing: {
    base: 29,
    "id-ID": 500000,
    "ja-JP": 4500,
  },
}
```

| API | Notes |
| --- | --- |
| `localPrice(plan)` | Returns the raw numeric value for the active locale (or `base`). Emits `data-local-price` for runtime re-hydration |
| `localPriceCurrency(plan)` | Returns the currency code for the active locale's price (falls back to `LOCALES[0].currency` = `USD`, sourced from the generated `active-locales-data.ts`). Emits `data-local-price` for runtime re-hydration |
| `formatLocalPrice(plan, options?)` | `formatCurrency(localPrice, localPriceCurrency)` |
| `convertCurrency(value, targetCurrency, options?)` | Convert from the current locale's currency to the target currency using `EXCHANGE_RATES` |
| `convertLocalPrice(plan, targetCurrency, options?)` | Convert the locale price to the target currency |
| `formatLocalPriceDiscounted(plan, multiplier, targetCurrency?, options?)` | Convert and apply a discount multiplier (`targetCurrency` is optional) |

The runtime mirror of `formatLocalPrice()` and `convertLocalPrice()` is `data-local-price` (with optional `data-discount` and `data-target-currency`). It reads the pricing payload directly from the attribute, applies the active locale, and falls back to `pricing.base` (in `USD`).

```njk
<span>{{ i18n.formatLocalPrice(plan) }}</span>
<span>{{ i18n.formatLocalPriceDiscounted(plan, 0.8) }}</span>
```

```html
<span data-local-price='{"base":29,"id-ID":500000}'></span>
```

Exchange rates are pulled from Frankfurter and committed to `generated/exchange-rates.ts`:

```bash
bun ./scripts/fetch-exchange-rates.ts    # uses the cache if it is < 24h old
bun ./scripts/fetch-exchange-rates.ts -- --force  # bypasses the cache
```

## Runtime

### Bootstrap

1. The Nunjucks build renders the page in the default locale and injects an inline `<script>` in `<head>`. For single-locale configs, this is minimal: `window.__SERVER_LOCALE__` + `window.__SAVED_LOCALE__`. For multi-locale, it also sets `window.__PAGE_ID__`, detects locale from localStorage/path, and sets `<html lang>`, `dir`, `data-script`.
2. `bootstrap.ts` is auto-injected into every page entry via `packages/page-system/page-inject-loader.cjs`. It imports `styles/main.css`, then loads Alpine + collapse/focus plugins in parallel.
3. **Single-locale path**: Bootstrap registers a stub Alpine i18n store (`{ current, languages: [], change: ()=>{} }`), then calls `Alpine.start()`. No i18next, no locale JSON fetch.
4. **Multi-locale path**: Bootstrap loads `registerI18nStore` + Alpine, registers store, calls `initIntl(locale)` BEFORE `Alpine.start()` so Alpine components have i18next ready. `initIntl` fetches `${BASE_PATH}assets/i18n/{pageID}/{locale}.json`, initializes i18next, schedules `translatePage()` + `updateFormattedElements()` via microtask queue.
5. `Alpine.start()` is deferred via `requestAnimationFrame` to reduce TBT (Total Blocking Time).
6. Font preloading (`preloadActiveFont`) runs as dynamic `import()`. `setupFontStackCSS`, `loadLanguageFonts`, `watchScriptAndLoadFont` deferred via `scheduler.postTask` (background priority).
7. Service worker registered (production only) via `navigator.serviceWorker.register()`.

### Window globals

Declared in `types/global.d.ts`:

| Global | Type | Purpose |
| --- | --- | --- |
| `window.__PAGE_ID__` | `keyof I18nPages` | Current page id (drives the page namespace) |
| `window.__SAVED_LOCALE__` | `LocaleCode` | Locale to activate on boot (from localStorage or path) |
| `window.__SERVER_LOCALE__` | `LocaleCode` | Default locale from server config |

### i18next configuration

`initIntl()` receives an optional `localeOverride` (from `__SAVED_LOCALE__` or `__SERVER_LOCALE__`). When `localeOverride` is set, it calls `i18next.init({ … })` directly with `lng` and `detection: undefined`. The `LanguageDetector` plugin is only used as a fallback when no override is provided (which does not happen in normal bootstrap).

```ts
{
  resources,                                  // flat namespace bundle (common, page, shared)
  supportedLngs: LOCALE_CODES,
  lng: savedLocale,
  ns: ['common', pageID],
  defaultNS: 'common',
  detection: localeOverride ? undefined : {
    order: ['localStorage', 'navigator'],
    caches: ['localStorage'],
    lookupLocalStorage: 'i18nextLocale',
  },
  interpolation: { escapeValue: false },
  saveMissing: false,
  parseMissingKeyHandler: (key) => `[${key}]`,
}
```

### Alpine store

Registered by `registerI18nStore()` (defined in `packages/i18n/runtime/store.ts`). The store reads from `localStorage` for `current` and writes to it on `change(code)`. The `change()` method delegates to an internal `changeLanguage()` function that orchestrates the full locale switch. On change it:

1. Saves `code` to `localStorage` under `i18nextLocale` and sets `this.current`.
2. Calls `changeLanguage(code)` which:
   - If i18next is not yet initialized, calls `initIntl(code)`.
   - Otherwise, calls `ensureLocaleData(code, pageID, m)` to fetch and load the locale's JSON bundle via `i18next.addResourceBundle()` (skips if already loaded).
   - Calls `i18next.changeLanguage(code)`.
   - Calls `scheduleTask(translatePage)`, `scheduleTask(updateFormattedElements)`, and `scheduleTask(updateI18nStoreLabels)` via the microtask queue for batched DOM updates.
3. Calls `updateDocumentAttributes(code)` to update `<html lang>`, `<html dir>`, `data-script`, and toggle `.is-rtl`.
4. Calls `refreshFonts()` which dynamically imports `packages/i18n/fonts/fonts.ts` and calls `setupFontStackCSS()` + `loadLanguageFonts()`.

```njk
<div x-data>
  <template x-for="lang in $store.i18n.languages" :key="lang.code">
    <button
      type="button"
      :aria-pressed="$store.i18n.current === lang.code"
      @click="$store.i18n.change(lang.code)"
    >
      <span x-text="lang.label"></span>
    </button>
  </template>
</div>
```

The `languages` array is derived from `LOCALES` filtered by `LOCALE_CODES`, with the human label produced by `getLocaleLabelCountry(code)` (e.g. `English (US)`, `简体中文 (CN)`).

## User Configuration

Two files control the i18n + font setup: `configs/i18n.ts` (locales) and `configs/fonts.ts` (fonts).

### Locale config (`configs/i18n.ts`)

```ts
// configs/i18n.ts
import { defineI18n } from '@i18n/config/define';

export const i18nConfig = defineI18n({
  defaultLocale: 'en-US',
  locales: ['id-ID', 'ja-JP', 'zh-Hans-CN', 'ar-SA'],
});
```

| Field | Type | Purpose |
| --- | --- | --- |
| `defaultLocale` | `LocaleCode` (BCP 47) | SSR locale, fallback target, default `getLocale()` |
| `locales` | `LocaleCode[]` (optional) | Active locales — pages generated, switcher shown, runtime data loaded. Defaults to `[defaultLocale]` only |

`defineI18n` validates the config at module load time. It checks for duplicate locales and validates each code against the master `LOCALE_CODES` from `data/locales.ts` (not the generated file) — so adding a new locale to the prod config works without a prior prod build.

### Font config (`configs/fonts.ts`)

```ts
// configs/fonts.ts
import '@fontsource-variable/inter/index.css';
// For local fonts: import '../assets/fonts/my-font.css';

import { defineFontStack } from '@i18n/config/define';

export const fontsConfig = defineFontStack({
  sans: {
    name: 'inter',
    family: 'Inter Variable',
  },
  // optional:
  // serif: { name: 'lora', family: 'Lora Variable' },
  // mono: { name: 'jetbrains-mono', family: 'JetBrains Mono Variable' },
  // custom keys also supported:
  // display: { name: 'clash-display', family: 'Clash Display' },
});
```

| Field | Type | Purpose |
| --- | --- | --- |
| `sans` | `FontConfig` | Sans-serif font (required). Maps to `--font-sans` |
| `serif` | `FontConfig` (optional) | Serif font. Maps to `--font-serif` |
| `mono` | `FontConfig` (optional) | Monospace font. Maps to `--font-mono` |
| `[key: string]` | `FontConfig` (optional) | Custom font families. Each key `X` maps to `--font-X`. Use with Tailwind v4 `font-(family-name:--font-X)` |

The CSS import at the top of the file controls which font stylesheet is bundled. For `@fontsource` packages, use the bare import (e.g. `@fontsource-variable/inter/index.css`). For local fonts, use a relative path (e.g. `../assets/fonts/my-font.css`) — the CSS file should contain `@font-face` declarations.

`defineFontStack` and `defineFont` validate font configs at module load time.

### Active locale selection (`locales`)

By default, only the `defaultLocale` is "active" — meaning only its pages are generated, its translations are loaded, and it appears in the navbar locale switcher. To enable additional locales, list them in the `locales` array:

```ts
export const i18nConfig = defineI18n({
  defaultLocale: 'en-US',
  locales: ['id-ID', 'ja-JP', 'zh-Hans-CN', 'ar-SA'],
});
```

**What "active" controls:**
- **Page generation**: Only active locale directories are generated in `dist/`
- **Locale switcher**: The navbar `$store.i18n.languages` only lists active locales (auto-hides when only 1 locale is active)
- **Runtime data**: `NATIVE_DIGITS_MAP` and other client-side data structures only include active locales
- **i18next `supportedLngs`**: Only active locales are passed to i18next

**What "active" does NOT control:**
- Helper functions (`getLanguageConfig`, `getCurrency`, `getDirection`, etc.) still resolve gracefully for any locale code via prefix matching and `LOCALES[0]` fallback. In production, the underlying `LOCALES`/`LOCALE_CODES` arrays are the filtered subset from `generated/active-locales-data.ts`; in dev/test they contain all 136 entries.
- Fallback chain resolution still works because unknown locale codes fall through to `['en-US']`.

**Default behavior**: If `locales` is omitted (or empty), only `defaultLocale` is active. This means a fresh project renders only `en-US` pages and the locale switcher is hidden.

The active locale list is accessible via `getActiveLocaleCodes()` and `getActiveLocales()` from `packages/i18n/engine/active-locales.ts`.

## Locale Configuration

All locale behavior is defined in the `packages/i18n/` package. The package combines pure data/formatters with the runtime engine (i18next init, DOM updates, Alpine store) — there is no per-locale flag in the templates. The locale data lives in `packages/i18n/data/locales.ts` (the `LOCALES` array). Every locale carries the fields below:

```ts
{
  code: 'en-US',
  language: 'en',
  region: 'US',
  flag: 'US',
  dir: 'ltr',
  currency: 'USD',
  numberingSystem: 'latn',
  nativeNumberingSystem: 'latn',
  nativeDigits: false,
  writingSystem: 'latin',
  timezone: 'America/New_York',
  timezoneOffset: -5,
  calendar: 'gregory',
  dateFormat: 'short',
  timeFormat: 'short',
  firstDayOfWeek: 0,
}
```

(Plural rules are no longer a per-locale field — `getPluralSuffix` uses the browser's `Intl.PluralRules` to infer the category from the active locale.)

For Chinese script variants, include `script`:

```ts
{
  code: 'zh-Hans-CN',
  language: 'zh',
  script: 'Hans',  // required for script variants
  region: 'CN',
  // … other fields
}
```

### Key fields

| Field | Purpose | Consumers |
| --- | --- | --- |
| `code` | BCP 47 tag; the only stable identifier | Everywhere |
| `language` | `LanguageCode` — used for fallback, digit map, ordinal/cardinal strategies | `getLanguageSubtag`, formatters |
| `script` | `ScriptCode` — required for Chinese script variants | Labeling, fallback |
| `region` | `RegionCode` | Display label, fallback target |
| `flag` | `FlagCode` — emoji or short code | `$store.i18n.languages[].flag` |
| `dir` | `'ltr'` \| `'rtl'` | `data-script` neighbor, `<html dir>` |
| `currency` | `CurrencyCode` | `getCurrency()`, default formatter currency |
| `numberingSystem` | `NumberingSystemCode` used for `Intl` by default | `Intl.NumberFormat` default |
| `nativeNumberingSystem` | `NumberingSystemCode` rendered when `nativeDigits: true` | `toNativeDigits()` |
| `nativeDigits` | Whether to render native digits by default | `toNativeDigits()` (no force) |
| `writingSystem` | `WritingSystemCode` — drives `<html data-script>` and font selection | `packages/template-engine/template.ts`, `bootstrap.ts` |
| `timezone` | IANA timezone | `Intl.DateTimeFormat` |
| `timezoneOffset` | Hours from UTC (decimal for half-hour zones) | Available for client logic |
| `calendar` | `CalendarCode` | `Intl.DateTimeFormat` |
| `dateFormat` / `timeFormat` | Default presets for date and time | Available for client logic |
| `firstDayOfWeek` | `0`–`6` (Sunday = 0) | Calendar widgets |

### Derived constants

| Constant | Source | Use |
| --- | --- | --- |
| `defaultLocale` (in `i18nConfig`) | `configs/i18n.ts` | SSR locale, fallback target |
| `LOCALE_STORAGE_KEY` | `'i18nextLocale'` (`packages/i18n/index.ts`) | `localStorage` key for the saved locale |
| `LOCALE_CODES` | `LOCALES.map(l => l.code)` | `supportedLngs`, sync targets |
| `LANGUAGE_CODE` / `SCRIPT_CODE` / `REGION_CODE` / `FLAG_CODE` | Same pattern as `LOCALE_CODES` | Type-safe code constants |
| `CURRENCY_CODE` / `TIMEZONE_CODE` / `CALENDAR_CODE` / `DIRECTION_CODE` | Same pattern as `LOCALE_CODES` | Type-safe code constants |
| `NUMBERING_SYSTEM_CODE` / `WRITING_SYSTEM` | Same pattern | Type-safe code constants |

The default currency anchor for `EXCHANGE_RATES` is hardcoded to `CURRENCY_CODE.USD` in `scripts/fetch-exchange-rates.ts` (the Frankfurter API base).

### Fallback chain

`getFallbackChain(locale)` in `packages/i18n/engine/helpers.ts` is called from both the build (`packages/template-engine/template.ts`) and the runtime (`packages/i18n/runtime/runtime.ts` → `fallbackLng`). The chain is:

1. If the locale is in `LOCALE_CODES`, return `[]` (no fallback needed).
2. If the language subtag matches a known locale, return `[matched, 'en-US']`.
3. Otherwise, return `['en-US']`.

All 136 configured locales are in `LOCALE_CODES`, so explicit fallback mapping is no longer needed — regional variants like `zh-Hans-SG` and `en-ZA` are fully configured with their own entries.

### Direction and font behavior

`data-script` on `<html>` is derived from `writingSystem` in both `packages/template-engine/template.ts` (initial render) and `packages/i18n/runtime/store.ts` (post-change). The writing system's `defaultFont` in `packages/i18n/data/writing-systems.ts` drives font selection — **changing the active locale re-runs `setupFontStackCSS()` to update `--font-sans`**, which in turn loads the appropriate `@fontsource(-variable)?/...` stylesheet.

Layout that must work in both LTR and RTL must use logical CSS properties. The RTL writing systems are: `arabic`, `hebrew`, `nko`, and `adlam`.

```css
.card {
  padding-inline: 1rem;
  margin-inline-start: 2rem;
  border-inline-end: 1px solid currentColor;
}
```

```css
/* Avoid */
.card {
  margin-left: 2rem;
  border-right: 1px solid currentColor;
}
```

### Translation strategy

The **94 locales added in June 2026** (12 European Latin + 33 across Turkic, Central Asian, South Asian Indic, Southeast Asian, Middle Eastern/Caucasus, Slavic/Baltic/Balkan + 49 additional covering African, additional European, Southeast Asian, South Asian, Middle Eastern, and regional English/Spanish/French/German variants) were translated per-locale manually with the following conventions:

- **Per-locale plural rules** (from CLDR): each locale uses its native plural categories instead of copying Indonesian's 6-variant template. For example, Dutch uses `one`/`other`; Czech uses `one`/`few`/`many`; Polish uses `one`/`few`/`many`; Arabic uses all six categories. Plural keys absent from the target locale are not generated.
- **Native script and numbering**: European Latin locales use `LATN` digits; South Asian Indic locales use their script's native numbering system (`BENG`, `GUJR`, `GURU`, `DEVA`, `TAMLDEC`, `TELU`, `KNDA`, `MLYM`) with `nativeDigits: true`. Cyrillic-script locales (`kk-KZ`, `ky-KG`, `tg-TJ`, `mn-MN`, `uk-UA`, `sr-RS`, `bg-BG`) use `CYRL`. RTL locales (`he-IL`, `fa-IR`, `pa-PK`, `uz-AF`) use `ARAB` or `HEBR` with appropriate native digit settings.
- **Locale-specific currency**: each locale is bound to its actual currency. Prices and currency formatting reflect the local currency.
- **Localized contact details**: phone numbers and addresses reflect the locale's primary city (e.g. `+90 312 123 4567` / "Ankara, Turkiye" for `tr-TR`).
- **Brand and technical terms kept in English**: "Technomira Tanzania", "Starter", "SOC 2", "GDPR", "Web Engine", "API", "SSO" remain in English across all locales.
- **HTML in values preserved**: tags like `<span class='text-violet-400'>...</span>` are kept intact for color highlighting in titles.
- **Placeholders preserved**: `{{count}}`, `{{year}}` are left as-is for runtime interpolation.

### Algorithmic numbering systems

Of the 12, five are bound to locales (`jpan`/`hans`/`hant`/`kore`/`cyrl`) and the rest are wired at the system level but not yet bound.

## Type Safety

`bun ./packages/i18n/cli/generate-types.ts` walks `locales/en-US` and writes `generated/i18n.d.ts`. The file contains:

- `I18nTranslationKeys` — discriminated union of all `'common:key'`, `'{page}:key'`, and `'{shared}:key'` strings.
- `I18nPages` — interface indexed by page id, with each value matching that page's JSON shape.
- `I18nShared` — interface indexed by shared locale name, with each value matching that locale's JSON shape.

`packages/i18n/runtime/runtime.ts` exports a typed `t(key: I18nTranslationKeys, vars?)` and `types/global.d.ts` references `I18nPages` for the window globals. This is the only way TypeScript can validate keys in client code.

> The type generator runs as part of `bun run build`. Run manually with `bun ./packages/i18n/cli/generate-types.ts`. Parity checking is integrated into `generate-types.ts` — locale key mismatches are treated as errors that fail the build (`process.exit(1)`). For a standalone parity report, use `bun ./packages/i18n/cli/check-parity.ts`.

## Best Practices

### Use `i18n.*` for user-facing copy

```njk
<h1>{{ i18n.t('home:hero.title') }}</h1>
```

### Use formatters instead of manual formatting

```njk
<span>{{ i18n.formatCurrency(plan.price, 'IDR') }}</span>
<time>{{ i18n.formatDate(page.published_at, { dateStyle: 'medium' }) }}</time>
```

### Use `raw` only when wrappers are not allowed

```njk
<meta name="description" content="{{ i18n.t('home:seo.description', {}, { raw: true }) }}">
```

### Use `attr()` for simple attributes

```njk
<input {{ i18n.attr('contact:form.email_placeholder', 'placeholder') }}>
```

### Use logical CSS for RTL-safe layouts

```njk
<div class="mbs-4 mie-2 pie-3 border-is"></div>
```

### Avoid hardcoded user-facing copy

```njk
<!-- Avoid -->
<h1>Welcome</h1>
```

### Avoid manual currency or date formatting

```njk
<!-- Avoid -->
<span>Rp {{ price }}</span>
```

### Template conventions

- Macros receive resolved text, not i18n keys (except `form-input.njk` which takes keys and receives `i18n`)
- String concatenation in Nunjucks: `~` everywhere (never `+`)
- Dynamic keys use `{% set %}` variables: `{% set ns = name %}{{ i18n.t(ns ~ ':heading') }}`
- No `i18n.text()` — use `i18n.t(key, vars, { raw: true })` for plain strings
- Error pages use `{% set %}` + `{% include %}` pattern (not macros) — macros have isolated scope and cannot access `i18n`, `url`, `icon`
- Error page quick links use string-encoded format (`'/|home,/features|features'`) split in partial — Nunjucks doesn't support object/array literals in macro args

### Error pages

Six error pages share two Nunjucks partials: `error-page.njk` (5 pages with CTA link) and `offline-page.njk` (reload button + cached pages hint). Each page uses `{% set %}` variables to configure icon, gradient colors, i18n namespace, and quick links:


Each error page has its own i18n namespace (`not-found`, `unauthorized`, `forbidden`, `server-error`, `maintenance`, `offline`) with locale files across all 136 locales. Error pages are detected by `page-meta.njk` via the `is_error_page` variable — they receive `noindex, nofollow` meta tags and no JSON-LD structured data.

Error page URLs are locale-dependent. Each error page's URL slug is defined per-locale in `SYSTEM_PAGE_SLUGS` (`packages/page-system/system-pages.ts`). For example, `not-found` renders as `/page-not-found` in English variants but `/halaman-tidak-ditemukan` in Indonesian. The `page_id` (`not-found`) stays constant for i18n namespacing — only the folder name and URL change. The service worker (`public/sw.js`, generated by `scripts/generate-sw.ts`) precaches all six error pages using these locale-specific URLs.

### Dev performance cache

`generateClientI18nScript()` in `packages/template-engine/template.ts` uses an in-memory cache (`i18nScriptCache`) to avoid regenerating the inline bootstrap script and locale JSON bundles on every dev page load. The cache is keyed by page name and validated by `hashLocales()`:

- `hashLocales()` computes an mtime-based hash across **all active locale files** (common + page + shared)
- On repeat page loads, if the hash matches the cached version, the script is returned from cache — skipping ~3,400 file I/O operations
- Cache invalidates automatically when any locale file changes (added, modified, deleted)
- Production builds always regenerate (fresh process, empty cache)

## Troubleshooting

### Quick diagnosis flowchart

```
Page looks wrong on a specific locale
            │
            ▼
  Is the right text showing?  ──No──►  check:parity, missing keys, fallback chain
            │ Yes
            ▼
  Is the right font showing?  ──No──►  See "Font issues" below
            │ Yes
            ▼
  Is the right number format?  ──No──►  Check i18n.format*() usage
            │ Yes
            ▼
  Is the right direction (RTL)?  ──No──►  Check logical CSS properties
```

### Common issues

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| `[home:hero.title]` is displayed | Missing key in the active resource | Add the key to the relevant locale and run `bun ./packages/i18n/cli/generate-types.ts` |
| SSR copy is correct but does not change after switching language | The output has no runtime data attribute | Use `i18n.t()` or a formatter helper |
| Attribute text does not change after switching language | `i18n.attr()` only renders the initial attribute | Use the `data-i18n-attr="attr:key"` manual pattern for runtime attribute updates |
| A number or currency does not change | The value is rendered manually | Use `i18n.formatNumber()` / `formatCurrency()` / `convertCurrency()` |
| Native digits disappear on language switch | The locale's `nativeDigits` is `false` so `toNativeDigits()` skips conversion | Switch to `i18n.t()` with a translation key, or `i18n.formatNumber(..., { nativeDigits: true })` |
| Price uses the wrong currency | The plan has no locale price and falls back to `base` | Add a locale-specific price or pass an explicit `targetCurrency` |
| RTL layout is broken | CSS uses physical direction properties | Use logical properties (`margin-inline-start`, `border-inline-end`, etc.) |
| Font does not switch with the locale | `setupFontStackCSS()` did not run | The `packages/i18n/runtime/store.ts` `change()` method dynamically imports `packages/i18n/fonts/fonts.ts` and calls `setupFontStackCSS()` + `loadLanguageFonts()` after i18next finishes; check the dynamic import isn't failing in DevTools |
| `<html data-script>` is `latin` for an Arabic locale | `data-script` is derived from `writingSystem`, not `numberingSystem` — for `ar-*` it is `arabic` by config | Verify `writingSystem` on the locale entry; this is by design |
| Build fails with type errors | JSON was edited and types are stale | Run `bun ./packages/i18n/cli/generate-types.ts` |
| `bun ./packages/i18n/cli/check-parity.ts` fails | Locale files have missing or extra keys | Align the key structure to match `locales/en-US/` |
| A shared locale is empty at runtime | The template doesn't use `i18n.t('{name}:...')` or the locale file is missing | Ensure the shared template uses the namespace prefix and the `{name}.json` file exists in `locales/{locale}/` |
| A locale button label is `id-ID` instead of `Indonesian (ID)` | `getLocaleLabelCountry()` did not match | Confirm the locale entry exists in `LOCALES` |

### Font issues

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| System font renders (Yu Gothic, Hiragino, etc.) instead of Noto Sans | `--font-sans` doesn't match the `font-family` registered by `@fontsource` | Compare `packages/i18n/data/writing-systems.ts` `defaultFont` with `@font-face` in `node_modules/@fontsource*/index.css`; update the config to match exactly |
| Font doesn't load on page refresh (only on language switch) | The locale's numbering system isn't in `FONT_CSS_PATHS` | Add the entry to `FONT_CSS_PATHS` in `packages/i18n/data/font-paths.ts` |
| "Module not found" during build | Missing `@fontsource*` package or wrong import path | Add to `package.json`; check `variable` vs non-variable path |
| First switch to a locale shows tofu briefly | First-time download of font CSS | Add to `FONT_CSS_PATHS` in `packages/i18n/data/font-paths.ts` for next refresh, or accept FOUT |

## Fonts

Font loading is part of the i18n system — it is driven by locale data, reacts to locale changes via `MutationObserver`, and lives at `packages/i18n/fonts/fonts.ts`.

### Loading strategy

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Page load (refresh)                          │
│                                                                     │
│  ┌──────────────────┐    ┌──────────────────┐    ┌──────────────┐  │
│  │  Inline script   │    │  bootstrap.ts    │    │  fonts.css   │  │
│  │  (template-engine/   │    │  setupFontStack  │    │  (only non-  │  │
│  │   template.ts)   │    │  CSS()           │    │   Latin WS)  │  │
│  └────────┬─────────┘    └────────┬─────────┘    └──────┬───────┘  │
│           │                       │                       │         │
│           ▼                       ▼                       ▼         │
│   <html lang="ja-JP">      --font-sans set        Noto Sans JP CSS│
│   data-script="cjk"        (for writing            (if ws !== latin│
│                             system)                   via dynamic   │
│                                                    import())        │
└─────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────┐
│                    User changes language                            │
│                                                                     │
│  Alpine store  ──►  <html lang="ar-SA">  ──►  MutationObserver     │
│   .change()          data-script="arabic"      detects change      │
│                                                                     │
│                                      ──►  loadFontForLang(ar-SA)   │
│                                           dynamic import of         │
│                                           noto-sans-arabic CSS     │
└─────────────────────────────────────────────────────────────────────┘
```

Two layers, one file (`packages/i18n/fonts/fonts.ts`):

1. **Static** (always loaded) — `import '@fontsource-variable/inter/index.css'` at top of `configs/fonts.ts`. Guarantees the default locale (`en-US` / Latin) renders correctly with system-ui fallback. For local fonts, replace this import with a relative path to your custom CSS.
2. **On-demand** (bootstrap + runtime) — `setupFontStackCSS()` sets `--font-sans` based on the writing system. `loadFontForLang()` dynamically imports the locale's font CSS via `import()` for non-Latin writing systems (`ws !== 'latin'`). This is called during bootstrap for the initial locale and on every locale change via `watchScriptAndLoadFont()` (MutationObserver on `<html lang>` and `<html data-script>`).

### Why does each CSS path appear twice in `ACTIVE_NOTO_SANS`?

Each entry in the generated `ACTIVE_NOTO_SANS` looks like:

```ts
jpan: {
  css: '@fontsource-variable/noto-sans-jp/index.css',
  loader: () => import('@fontsource-variable/noto-sans-jp/index.css'),
},
```

The string appears once in `css` (for reference) and once inside the loader's `import()` call. **The duplication is required for code splitting.**

`import(constVariable)` (using a top-level const for the path) makes Rsbuild/Rspack merge all 26 fonts into one CSS chunk. The literal string in the loader's `import()` is what lets the bundler create per-font chunks that are loaded on demand.

If you add a new entry, copy the string in both places — they must match exactly.

### Adding a new writing system

Example: adding Bengali (`beng`).

1. **Add the package to `package.json`:**
   ```json
   "@fontsource-variable/noto-sans-bengali": "^5.2.10"
   ```

2. **Add the entry to `WRITING_SYSTEMS`** in `packages/i18n/data/writing-systems.ts`:
   ```ts
   {
     code: 'bengali',
     name: 'Bengali Script',
     nameId: 'Skrip Bengali',
     description: 'Bengali script used in Bengali and Assamese languages',
     languages: [],
     numberingSystems: ['beng'],
     direction: DIRECTION_CODE.LTR,
     defaultFont: 'Noto Sans Bengali Variable',
   }
   ```

3. **Add the entry to `FONT_CSS_PATHS`** in `packages/i18n/data/font-paths.ts` (the generator will produce both `css` and `loader` with the literal string):
   ```ts
   beng: {
     css: '@fontsource-variable/noto-sans-bengali/index.css',
     loader: () => import('@fontsource-variable/noto-sans-bengali/index.css'),
   },
   ```
   The `SCRIPT_FONT_LOADERS` and `SCRIPT_FONT_CSS` maps are derived from this automatically.

4. **Add at least one locale** with `nativeNumberingSystem: NUMBERING_SYSTEM_CODE.BENG` in `packages/i18n/data/locales.ts`.

5. **Run `bun run build`** to verify the font bundle is created in `dist/assets/fonts/`.
