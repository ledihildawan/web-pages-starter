# i18n

The internationalization system in this project is built around a single source of truth for both copy and behavior. Source translations live as JSON5 under `src/locales`, while the full i18n runtime (locale data, formatters, i18next init, Alpine store, Nunjucks template) lives in `src/scripts/lib/i18n/`. The same keys are consumed by the Nunjucks template helpers for server-side rendering and by an i18next runtime that updates the DOM in place when the user switches language.

This document is the canonical reference for the system. It is accurate as of the file's "Last updated" line at the bottom and tracks the actual implementation, not aspirational behavior.

## Source Translations

### File layout

```txt
src/locales/
  id-ID/                       # default locale
    common.json5               # site-wide copy
    home.json5                 # page-specific copy
    pricing.json5
    components/
      cta.json5                # reusable component copy
  en-US/
    common.json5
    home.json5
    ...
```

Both `.json5` and `.json` are supported. A `.json` file is picked up automatically when the `.json5` equivalent is absent.

### Namespace resolution

The template key prefix determines the i18next namespace at runtime.

| Template key | Runtime namespace | Source file |
| --- | --- | --- |
| `site_name` | `common:site_name` | `src/locales/{locale}/common.json5` |
| `page.hero.title` | `{page}:hero.title` | `src/locales/{locale}/{page}.json5` |
| `comp.cta.title` | `components/cta:title` | `src/locales/{locale}/components/cta.json5` |

The `i18next` default namespace is `common`. The page namespace is registered on `i18next.init` along with any namespaces for components used by the current page (the component list is detected by `getUsedComponents()` in `template.ts`).

### Plural keys

i18next pluralization is keyed by suffix. The suffix for a given count is computed by `getPluralSuffix(count, locale?)` using the browser's `Intl.PluralRules` (no per-locale config required — categories are inferred from the active locale automatically):

| `Intl.PluralRules` category | i18next key suffix |
| --- | --- |
| `zero` | `_zero` |
| `one` | `_one` |
| `two` | `_two` |
| `few` | `_few` |
| `many` | `_many` |
| `other` | `_other` |

```json5
// src/locales/id-ID/home.json5
{
  hero: {
    title: "Build multilingual pages faster",
    subtitle: "One translation source for server rendering and runtime switching.",
  },
  stats: {
    projects_one: "{{count}} proyek",
    projects_other: "{{count}} proyek",
    projects_zero: "0 proyek",
    projects_two: "{{count}} proyek",
    projects_few: "{{count}} proyek",
    projects_many: "{{count}} proyek",
  },
}
```

```njk
{{ i18n.t('page.hero.title') }}
{{ i18n.plural('page.stats.projects', 7) }}
```

### Variables

Variables use the `{{name}}` syntax and are interpolated by both the SSR `resolve()` and i18next at runtime.

```json5
{ welcome: "Hello {{name}}" }
```

```njk
{{ i18n.t('welcome', { name: 'Dinda' }) }}
```

## Template API

The Nunjucks `i18n` object is created in `src/scripts/lib/i18n/template.ts` and is available to every template and partial that extends `main.njk`.

### Translation API

| API | Returns | Notes |
| --- | --- | --- |
| `i18n.t(key, vars?, options?)` | Wrapped HTML | `data-i18n` span; refreshes on language change |
| `i18n.text(key, vars?)` | Plain string | No wrapper; use for non-HTML contexts |
| `i18n.html(key, vars?)` | Raw locale string | Returns the resolved string with no wrapper (useful inside attribute values) |
| `i18n.attr(key, attrName, vars?)` | `attrName="…"` | Emits a complete attribute, e.g. `alt="…"` |
| `i18n.plural(key, count, vars?, options?)` | Wrapped HTML | `data-i18n-plural` + `data-i18n-count`; refreshes on language change |
| `i18n.getPluralSuffix(count)` | `_one` \| `_other` \| `_few` \| `_many` \| `_zero` \| `_two` | Useful for direct key construction |

`i18n.t()` options:

| Option | Effect |
| --- | --- |
| `raw: true` | Return a string without the wrapper span |
| `native: true` | Force native digits in the output (`data-force-native`) |
| `universal: true` | Force Latin digits (`0-9`) in the output (`data-force-universal`) |
| `className: '…'` | Add a class to the wrapper span |

`i18n.plural()` options: `raw`, `className`. The `count` is automatically added to the `vars` and the correct suffix is appended to the key for SSR.

Example output:

```html
<span data-i18n="home:hero.title">Build multilingual pages faster</span>
```

### Formatting API

Every formatter renders a `<span>` with one or more `data-format-*` attributes. `updateFormattedElements()` walks the DOM and replaces the inner text whenever the active locale changes. Pass `{ raw: true }` to receive a plain string.

| API | Data attribute | Notes |
| --- | --- | --- |
| `formatNumber(value, options?)` | `data-format-number` | `Intl.NumberFormat` |
| `formatCurrency(value, currency, options?)` | `data-format-currency` + `data-target-currency` | Currency code is required |
| `formatPercent(value, options?)` | `data-format-percent` | `0.85` → `85%` |
| `formatBytes(bytes, decimals = 1, options?)` | `data-format-bytes` + `data-bytes-decimals` | Uses binary units (`KiB`-style but labeled `KB`) |
| `formatDuration(seconds, options?)` | `data-format-duration` | Combines `Intl.RelativeTimeFormat` and `Intl.ListFormat` |
| `formatDate(date, options?)` | `data-format-date` | Uses locale timezone and calendar |
| `formatTime(date, preset?, options?)` | `data-format-time` + `data-time-preset` | `preset` is `'short'` \| `'medium'` \| `'long'` \| `'full'` |
| `formatDateTime(date, options?)` | `data-format-datetime` | Combines date and time |
| `formatOrdinal(value, options?)` | `data-format-ordinal` | Manual strategies for `id`, `ja`, `ar`; `Intl.PluralRules` + English suffix map for all other locales |
| `formatCardinal(value, options?)` | `data-format-cardinal` | Manual strategies for `id`, `ja`, `zh`, `ar`; otherwise `Intl.NumberFormat` (no spellout — `Intl` has no API for cardinal spellout). `options.gender: 'masculine' \| 'feminine'` only affects the Arabic strategy. |
| `formatScientific(value, options?)` | `data-format-scientific` | `Intl` scientific notation with native digit fallbacks |
| `formatAbbreviated(value, options?)` | `data-format-abbreviated` | Compact form such as `15.3K` |
| `formatList(items, options?)` | `data-format-list` + `data-items` | `Intl.ListFormat` |
| `formatUnit(value, unit, options?)` | `data-format-unit` + `data-unit` | `Intl.NumberFormat` with `style: 'unit'` |
| `formatRelativeTime(value, options)` | `data-relative-time` + `data-unit` + `data-numeric` | `options.unit` is required; `options.numeric` is `'always'` \| `'auto'` |
| `nativeDigits(text, force?, options?)` | `data-native-digits` + `data-force` | Initial conversion + refreshed at runtime |
| `pluralize(word, count?, inclusive?, options?)` | `data-pluralized` + `data-count` + `data-inclusive` | Initial English pluralization; refreshed at runtime |
| `singularize(word, options?)` | `data-singularized` | Initial English singularization; refreshed at runtime |
| `getPluralSuffix(count)` | – | Returns the suffix string |

Common options supported across the numeric formatters:

| Option | Effect |
| --- | --- |
| `raw: true` | Return a string without a wrapper span |
| `className: '…'` | Add a class to the wrapper span |
| `nativeDigits: true` | Render in the locale native numbering system |
| `numberingSystem: 'latn' \| 'arab' \| 'deva' \| …` | Override the `Intl` numbering system |
| `style: 'currency' \| 'percent' \| 'unit' \| …` | Standard `Intl` options |

Any `Intl.NumberFormatOptions` or `Intl.DateTimeFormatOptions` key is forwarded to `Intl.*Format` so things like `maximumFractionDigits`, `dateStyle: 'long'`, and `timeZoneName: 'short'` work without explicit support here.

```njk
{{ i18n.formatDate('2026-06-04', { dateStyle: 'long' }) }}
{{ i18n.formatNumber(1200, { maximumFractionDigits: 0 }) }}
{{ i18n.formatTime(page.published_at, 'medium') }}
```

### Grammatical gender (Arabic cardinals)

Arabic cardinal numerals change form based on the gender of the noun they modify. The `formatCardinal()` formatter accepts an optional `gender` field for Arabic locales; it is ignored for `id`, `ja`, `zh`, and the `Intl` fallback. Default is `'masculine'`.

```njk
{{ i18n.formatCardinal(2) }}
{# → "اِثنان" (masculine, default) #}

{{ i18n.formatCardinal(2, { gender: 'feminine' }) }}
{# → "اِثنتان" #}

{{ i18n.formatCardinal(99, { gender: 'feminine' }) }}
{# → "تِسع وتِسْعون" #}
```

The Arabic strategy covers `0`–`999,999,999,999` (units, teens, tens, hundreds, thousands, millions). Larger numbers fall back to the raw digit string. To extend the strategy (or add support for `Intl.PluralRules` integration with `formatCurrency`), edit `src/scripts/lib/i18n/formatters.ts`.

### Pricing API

Pricing plans use a regional pricing object. `pricing.base` is the USD fallback. Add a per-locale entry when local pricing exists.

```json5
{
  pricing: {
    base: 29,
    "id-ID": 500000,
    "ja-JP": 4500,
  },
}
```

| API | Notes |
| --- | --- |
| `localPrice(plan)` | Returns the raw numeric value for the active locale (or `base`) |
| `localPriceCurrency(plan)` | Returns the currency code for the active locale's price (or `USD`) |
| `formatLocalPrice(plan, options?)` | `formatCurrency(localPrice, localPriceCurrency)` |
| `convertCurrency(value, targetCurrency?, options?)` | Convert from the current locale's currency to the target currency using `EXCHANGE_RATES` |
| `convertLocalPrice(plan, targetCurrency?, options?)` | Convert the locale price to the target currency |
| `formatLocalPriceDiscounted(plan, multiplier, targetCurrency?, options?)` | Convert and apply a discount multiplier |

The runtime mirror of `formatLocalPrice()` and `convertLocalPrice()` is `data-local-price` (with optional `data-discount` and `data-target-currency`). It reads the pricing payload directly from the attribute, applies the active locale, and falls back to `pricing.base` (in `USD`).

```njk
<span>{{ i18n.formatLocalPrice(plan) }}</span>
<span>{{ i18n.formatLocalPriceDiscounted(plan, 0.8) }}</span>
```

```html
<span data-local-price='{"base":29,"id-ID":500000}'></span>
```

Exchange rates are pulled from Frankfurter and committed to `generated/exchange-rates.ts`:

```bash
bun run fetch:rates        # uses the cache if it is < 24h old
bun run fetch:rates:force  # bypasses the cache
```

## Runtime

### Bootstrap

1. The Nunjucks build renders the page in the default locale and injects an inline `<script>` in `<head>`. The script reads the saved locale from `localStorage`, the path (`/{lang}/...`), or the default, and writes `window.__PAGE_ID__`, `window.__SAVED_LOCALE__`, `window.__SERVER_LOCALE__`, and `window.__USED_COMPONENTS__`. It also sets `<html lang>`, `<html dir>`, `data-script`, and toggles `.is-rtl` to avoid a flash.
2. `src/scripts/main.ts` loads vendors (Alpine, plugins), then app modules (`initIntl`, `registerI18nStore`, navbar) in parallel, then `Alpine.start()`.
3. `initIntl()` initializes i18next with the inline resources, sets the language to the saved locale, and calls `translatePage()` and `updateFormattedElements()`.
4. After `initIntl()`, `main.ts` runs `setupFontStackCSS()` and triggers the per-locale font loaders.

### Window globals

Declared in `src/global.d.ts`:

| Global | Type | Purpose |
| --- | --- | --- |
| `window.__PAGE_ID__` | `keyof I18nPages` | Current page id (drives the page namespace) |
| `window.__USED_COMPONENTS__` | `(keyof I18nComponents)[]` | Components the page imports, used to namespace component locales |
| `window.__SAVED_LOCALE__` | `LocaleCode` | Locale to activate on boot (from localStorage or path) |
| `window.__SERVER_LOCALE__` | `LocaleCode` | Default locale from server config |

### i18next configuration

`initIntl()` calls `i18next.init({ … })` directly (without `LanguageDetector`) when `localeOverride` is provided (e.g., from `__SAVED_LOCALE__`). Otherwise it uses `i18next.use(LanguageDetector).init({ … })` to detect browser language on first visit.

```ts
{
  resources,
  fallbackLng: getFallbackForLocale,  // function: returns ordered fallback chain
  supportedLngs: LOCALE_CODES,
  ns: ['common', pageID],            // plus `components/{name}` for each used component
  defaultNS: 'common',
  detection: {
    order: ['localStorage', 'navigator'],
    caches: ['localStorage'],
    lookupLocalStorage: 'i18nextLocale',
  },
  interpolation: { escapeValue: false },
  saveMissing: false,
  parseMissingKeyHandler: (key) => `[${key}]`,  // dev-only warning on first miss
}
```

### Alpine store

Registered by `registerI18nStore()`. The store reads from `localStorage` for `current` and writes to it on `change(code)`. On change it:

1. Saves `code` to `localStorage` under `i18nextLocale`.
2. Calls `i18next.changeLanguage(code)`.
3. Updates `<html lang>`, `<html dir>`, `data-script`, and toggles `.is-rtl`.
4. Calls `translatePage()` and `updateFormattedElements()`.
5. Calls `updateI18nStoreLabels()` to refresh language labels.
6. Calls `setupFontStackCSS()` + `loadLanguageFonts()` (from `lib/i18n/fonts.ts`, dynamically imported) to repaint `--font-primary` and load the locale's font.

```njk
<div x-data>
  <template x-for="lang in $store.i18n.languages" :key="lang.code">
    <button
      type="button"
      :aria-pressed="$store.i18n.current === lang.code"
      @click="$store.i18n.change(lang.code)"
    >
      <span x-text="lang.label"></span>
    </button>
  </template>
</div>
```

The `languages` array is derived from `LOCALES` filtered by `LOCALE_CODES`, with the human label produced by `getLocaleLabelCountry(code)` (e.g. `English (US)`, `简体中文 (CN)`).

## User Configuration

The only file a user needs to touch to customize the i18n setup is `src/configs/i18n.ts`:

```ts
// src/configs/i18n.ts
import { defineI18n, defineFont } from '../scripts/lib/i18n/i18n-config';
import type { I18nConfig } from '../scripts/lib/i18n/types';

export const i18nConfig: I18nConfig = defineI18n({
  defaultLocale: 'id-ID',
  fonts: {
    primary: defineFont({
      name: 'inter',                 // @fontsource package name
      family: 'Inter',               // CSS font-family value
    }),
    // optional:
    // secondary: defineFont({ name: 'roboto', family: 'Roboto' }),
    // monospace: defineFont({ name: 'jetbrains-mono', family: 'JetBrains Mono' }),
  },
});
```

Two fields only:

| Field | Type | Purpose |
| --- | --- | --- |
| `defaultLocale` | `LocaleCode` (BCP 47) | SSR locale, fallback target, default `getLocale()` |
| `fonts.primary` | `FontConfig` | Primary font: `@fontsource` package name + CSS family |
| `fonts.secondary` | `FontConfig` (optional) | Secondary font |
| `fonts.monospace` | `FontConfig` (optional) | Monospace font |

`defineI18n` and `defineFont` validate the config at module load time. A missing field or invalid `name` throws an error immediately, not at first render. TypeScript also enforces the shape — invalid font names are rejected at compile time.

The remaining config — `LOCALES`, `LOCALE_CODES`, the 54-locale `data.ts` array — is engine data, not user config. Add locales there if you need more than the bundled 54.

## Locale Configuration

All locale behavior is defined in the `src/scripts/lib/i18n/` package. The package combines pure data/formatters with the runtime engine (i18next init, DOM updates, Alpine store, Nunjucks template) — there is no per-locale flag in the templates. The locale data lives in `src/scripts/lib/i18n/data.ts` (the `LOCALES` array). Every locale carries the fields below:

```ts
{
  code: 'id-ID',
  language: 'id',
  region: 'ID',
  flag: 'ID',
  dir: 'ltr',
  currency: 'IDR',
  numberingSystem: 'latn',
  nativeNumberingSystem: 'latn',
  nativeDigits: false,
  writingSystem: 'latin',
  timezone: 'Asia/Jakarta',
  timezoneOffset: 7,
  calendar: 'gregory',
  dateFormat: 'short',
  timeFormat: 'short',
  firstDayOfWeek: 1,
}
```

(Plural rules are no longer a per-locale field — `getPluralSuffix` uses the browser's `Intl.PluralRules` to infer the category from the active locale.)

For Chinese script variants, include `script`:

```ts
{
  code: 'zh-Hans-CN',
  language: 'zh',
  script: 'Hans',  // required for script variants
  region: 'CN',
  // … other fields
}
```

### Key fields

| Field | Purpose | Consumers |
| --- | --- | --- |
| `code` | BCP 47 tag; the only stable identifier | Everywhere |
| `language` | `LanguageCode` — used for fallback, digit map, ordinal/cardinal strategies | `getLanguageSubtag`, formatters |
| `script` | `ScriptCode` — required for Chinese script variants | Labeling, fallback |
| `region` | `RegionCode` | Display label, fallback target |

| `currency` | `CurrencyCode` | `getCurrency()`, default formatter currency |

| `nativeDigits` | Whether to render native digits by default | `toNativeDigits()` (no force) |

| Constant | Source | Use |

### Fallback chain

`getFallbackChain(locale)` in `src/scripts/lib/i18n/helpers.ts` is called from both the build (`template.ts`) and the runtime (`runtime.ts` → `fallbackLng`). The chain is:

1. If the locale is in `LOCALE_CODES`, return `[]` (no fallback needed).
2. If the language subtag matches a known locale, return `[matched, 'en-US']`.
3. Otherwise, return `['en-US']`.

All 54 configured locales are in `LOCALE_CODES`, so explicit fallback mapping is no longer needed — regional variants like `zh-Hans-SG` and `en-ZA` are fully configured with their own entries.

### Direction and font behavior

`data-script` on `<html>` is derived from `writingSystem` in both `template.ts` (initial render) and `store.ts` (post-change). The writing system's `defaultFont` in `writing-systems.ts` drives font selection — **changing the active locale re-runs `setupFontStackCSS()` to update `--font-primary`**, which in turn loads the appropriate `@fontsource(-variable)?/...` stylesheet.

Layout that must work in both LTR and RTL must use logical CSS properties:

```css
.card {
  padding-inline: 1rem;
  margin-inline-start: 2rem;
  border-inline-end: 1px solid currentColor;
}
```

```css
/* Avoid */
.card {
  margin-left: 2rem;
  border-right: 1px solid currentColor;
}
```

### Translation strategy

The 12 European Latin locales added in June 2026 were translated per-locale manually with the following conventions:

- **Per-locale plural rules** (from CLDR): each locale uses its native plural categories instead of copying Indonesian's 6-variant template. For example, Dutch uses `one`/`other`; Czech uses `one`/`few`/`many`; Polish uses `one`/`few`/`many`. Plural keys absent from the target locale are not generated.
- **Latin numbering system**: all 12 new locales use `NUMBERING_SYSTEM_CODE.LATN` (digits `0-9`). None of these languages have a separate native digit system, so no `nativeDigits` flag is set.
- **Locale-specific currency**: each locale is bound to its actual currency (`NOK`, `SEK`, `DKK`, `CZK`, `HUF`, `RON`, `PLN`, `EUR`). Prices and currency formatting reflect the local currency.
- **Localized contact details**: phone numbers and addresses reflect the locale's primary city (e.g. `+31 20 1234 5678` / "Amsterdam, Nederland" for `nl-NL`).
- **Brand and technical terms kept in English**: "Technomira Tanzania", "Starter", "SOC 2", "GDPR", "Web Engine", "API", "SSO" remain in English across all locales.
- **HTML in values preserved**: tags like `<span class='text-violet-400'>...</span>` are kept intact for color highlighting in titles.
- **Placeholders preserved**: `{{count}}`, `{{year}}` are left as-is for runtime interpolation.

### Algorithmic numbering systems

Of the 11, four are bound to locales (`jpan`/`hans`/`hant`/`kore`/`cyrl`) and the rest are wired at the system level but not yet bound.

If a locale exists in `LOCALES` but its files are still copies of `id-ID`:

## Type Safety

`bun run gen:i18n` walks `src/locales/id-ID` and writes `generated/i18n.d.ts`. The file contains:

- `I18nTranslationKeys` — discriminated union of all `'common:key'`, `'{page}:key'`, and `'components/{name}:key'` strings.
- `I18nCommon` — interface matching the shape of `common.json5`.
- `I18nPages` — interface indexed by page id, with each value matching that page's JSON5 shape.
- `I18nComponents` — interface indexed by component name, with each value matching that component's JSON5 shape.

`src/scripts/lib/i18n/runtime.ts` exports a typed `t(key: I18nTranslationKeys, vars?)` and `src/global.d.ts` references `I18nCommon`, `I18nPages`, and `I18nComponents` for the window globals. This is the only way TypeScript can validate keys in client code.

> ℹ️ The generator also runs a parity diff (`compareLocaleParity` in `tools/generate-i18n.ts`) but the throw is currently commented out. Use `bun run check:parity` for the authoritative parity report.

### Add page copy

1. Edit `src/locales/id-ID/{page}.json5` (the file is created by `gen:page`).
2. Mirror the key structure in every other locale under `src/locales/`.
3. Use `{{ i18n.t('page.section.key') }}` in the template.
4. Run `bun run gen:i18n`.
5. Run `bun run check:parity`.

### Add reusable component copy

1. Create or edit `src/locales/id-ID/components/{name}.json5`.
2. Include the component in the page template (`{% include 'components/{name}.njk' %}`). `getUsedComponents()` picks it up automatically.
3. Use `{{ i18n.t('comp.{name}.key') }}`.
4. Run `bun run gen:i18n`.

### Add a new locale

1. Add the entry to `LOCALES` in `src/scripts/lib/i18n/data.ts`.
2. Add the language, region, currency, flag, timezone, and (if new) writing system + numbering system entries to the matching per-concern file under `src/scripts/lib/i18n/`.
3. Run `bun run sync:locales` to copy the default structure.
4. Translate the new JSON5 files. Apply per-locale plural rules and keep brand/technical terms in English. See [Translation Coverage](#translation-coverage) for the conventions used in the 12 European Latin locales added in June 2026.
5. Run `bun run gen:i18n` and `bun run check:parity`.
6. Run `bun run build` to confirm the locale renders end-to-end.

## Best Practices

### Use `i18n.*` for user-facing copy

```njk
<h1>{{ i18n.t('page.hero.title') }}</h1>
```

### Use formatters instead of manual formatting

```njk
<span>{{ i18n.formatCurrency(plan.price, 'IDR') }}</span>
<time>{{ i18n.formatDate(page.published_at, { dateStyle: 'medium' }) }}</time>
```

### Use `raw` only when wrappers are not allowed

```njk
<meta name="description" content="{{ i18n.t('page.seo.description', {}, { raw: true }) }}">
```

### Use `attr()` for simple attributes

```njk
<input {{ i18n.attr('page.form.email_placeholder', 'placeholder') }}>
```

### Use logical CSS for RTL-safe layouts

```njk
<div class="mbs-4 mie-2 pie-3 border-is"></div>
```

### Avoid hardcoded user-facing copy

```njk
<!-- Avoid -->
<h1>Welcome</h1>
```

### Avoid manual currency or date formatting

```njk
<!-- Avoid -->
<span>Rp {{ price }}</span>
```

## Troubleshooting

### Quick diagnosis flowchart

```
Page looks wrong on a specific locale
            │
            ▼
  Is the right text showing?  ──No──►  check:parity, missing keys, fallback chain
            │ Yes
            ▼
  Is the right font showing?  ──No──►  See "Font issues" below
            │ Yes
            ▼
  Is the right number format?  ──No──►  Check i18n.format*() usage
            │ Yes
            ▼
  Is the right direction (RTL)?  ──No──►  Check logical CSS properties
```

### Common issues

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| `[home:hero.title]` is displayed | Missing key in the active resource | Add the key to the relevant locale and run `bun run gen:i18n` |
| SSR copy is correct but does not change after switching language | The output has no runtime data attribute | Use `i18n.t()` or a formatter helper |
| Attribute text does not change after switching language | `i18n.attr()` only renders the initial attribute | Use the `data-i18n-attr="attr:key"` manual pattern for runtime attribute updates |
| A number or currency does not change | The value is rendered manually | Use `i18n.formatNumber()` / `formatCurrency()` / `convertCurrency()` |
| Native digits disappear on language switch | The locale's `nativeDigits` is `false` so `toNativeDigits()` skips conversion | Switch to `i18n.t()` with a translation key, or `i18n.formatNumber(..., { nativeDigits: true })` |
| Price uses the wrong currency | The plan has no locale price and falls back to `base` | Add a locale-specific price or pass an explicit `targetCurrency` |
| RTL layout is broken | CSS uses physical direction properties | Use logical properties (`margin-inline-start`, `border-inline-end`, etc.) |
| Font does not switch with the locale | `setupFontStackCSS()` did not run | The `lib/i18n/store.ts` `change()` method dynamically imports `fonts.ts` and calls `setupFontStackCSS()` + `loadLanguageFonts()` after i18next finishes; check the dynamic import isn't failing in DevTools |
| `<html data-script>` is `latin` for an Arabic locale | `data-script` is derived from `writingSystem`, not `numberingSystem` — for `ar-*` it is `arabic` by config | Verify `writingSystem` on the locale entry; this is by design |
| Build fails with type errors | JSON5 was edited and types are stale | Run `bun run gen:i18n` |
| `bun run check:parity` fails | Locale files have missing or extra keys | Run with `--verbose` and align the key structure |
| A component translation is empty at runtime | The component is not included by the page | `getUsedComponents()` only loads locales for components the template imports; add the include |
| A locale button label is `id-ID` instead of `Indonesian (ID)` | `getLocaleLabelCountry()` did not match | Confirm the locale entry exists in `LOCALES` |

### Font issues

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| System font renders (Yu Gothic, Hiragino, etc.) instead of Noto Sans | `--font-primary` doesn't match the `font-family` registered by `@fontsource` | Compare `writing-systems.ts` `defaultFont` with `@font-face` in `node_modules/@fontsource*/index.css`; update the config to match exactly |
| Font doesn't load on page refresh (only on language switch) | The locale's numbering system isn't in `NOTO_SANS` | Add the entry to the `NOTO_SANS` map in `lib/i18n/fonts.ts` |
| "Module not found" during build | Missing `@fontsource*` package or wrong import path | Add to `package.json`; check `variable` vs non-variable path |
| First switch to a locale shows tofu briefly | First-time download of font CSS | Add to `NOTO_SANS` in `lib/i18n/fonts.ts` for next refresh, or accept FOUT |

## Fonts

Font loading is part of the i18n system — it is driven by locale data, reacts to locale changes via `MutationObserver`, and lives at `src/scripts/lib/i18n/fonts.ts`.

### Hybrid loading strategy

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Page load (refresh)                          │
│                                                                     │
│  ┌──────────────────┐    ┌──────────────────┐    ┌──────────────┐  │
│  │  Inline script   │    │  Static import   │    │  Preload     │  │
│  │  (template.ts)   │    │  (fonts.ts)      │    │  (preload)   │  │
│  └────────┬─────────┘    └────────┬─────────┘    └──────┬───────┘  │
│           │                       │                       │         │
│           ▼                       ▼                       ▼         │
│   <html lang="ja-JP">      Noto Sans Latin         <link rel=preload>│
│   data-script="cjk"        (always loaded)         noto-sans-jp.css │
│   --font-primary set                                (high priority)│
└─────────────────────────────────────────────────────────────────────┘
                                     │
                                     ▼
┌─────────────────────────────────────────────────────────────────────┐
│                    User changes language                            │
│                                                                     │
│  Alpine store  ──►  <html lang="ar-SA">  ──►  MutationObserver     │
│   .change()          data-script="arabic"      detects change      │
│                                                                     │
│                                      ──►  Dynamic import of        │
│                                           noto-sans-arabic CSS     │
└─────────────────────────────────────────────────────────────────────┘
```

Three layers, one file (`lib/i18n/fonts.ts`):

1. **Static** (always loaded) — `import '@fontsource-variable/inter/index.css'` at top of `fonts.ts`. Guarantees the default locale (`id-ID` / Latin) renders correctly.
2. **Preload** (active locale on refresh) — `preloadActiveFont()` adds `<link rel="preload">` for the current locale's font. Browser fetches in parallel with HTML parsing.
3. **On-demand** (locale changes at runtime) — `watchScriptAndLoadFont()` runs `MutationObserver` on `<html lang>` and `<html data-script>`. On change, dynamically imports the new font's CSS.

### File map (font-related)

| Concern | Where |
|---------|-------|
| Font config (name, family) | `src/configs/i18n.ts` (`i18nConfig.fonts`) |
| Font builders + types | `src/scripts/lib/i18n/i18n-config.ts` + `src/scripts/lib/i18n/types.ts` |

| Per-script Noto Sans CSS paths + dynamic loaders | `src/scripts/lib/i18n/fonts.ts` (`NOTO_SANS`) |
| Public API (called by `main.ts` and `lib/i18n/store.ts`) | `src/scripts/lib/i18n/fonts.ts` |

### Why is each CSS path written twice in `NOTO_SANS`?

Each entry in `NOTO_SANS` looks like:

```ts
jpan: {
  css: '@fontsource-variable/noto-sans-jp/index.css',
  loader: () => import('@fontsource-variable/noto-sans-jp/index.css'),
},
```

The string appears once in `css` (for the preload link) and once inside the loader's `import()` call. **The duplication is required for code splitting.**

`import(constVariable)` (using a top-level const for the path) makes Rsbuild/Rspack merge all 21 fonts into one CSS chunk (~226 kB loaded on the initial page). The literal string in the loader's `import()` is what lets the bundler create per-font chunks that are loaded on demand.

If you add a new entry, copy the string in both places — they must match exactly.

### Font name resolution

CSS uses a `--font-primary` variable that must match the `font-family` registered by `@fontsource`. The font family is defined per writing system in `writing-systems.ts` (`defaultFont`):

| Writing System | `--font-primary` value | Package |
|---------------|------------------------|---------|
| `latin` | `"Inter Variable"` | `@fontsource-variable/inter` |
| `cjk` | `"Noto Sans JP/SC/TC/KR Variable"` | `@fontsource-variable/noto-sans-jp/sc/tc/kr` |
| `arabic` | `"Noto Sans Arabic Variable"` | `@fontsource-variable/noto-sans-arabic` |
| `devanagari` | `"Noto Sans Devanagari"` | `@fontsource/noto-sans-devanagari` |
| `thai` | `"Noto Sans Thai Variable"` | `@fontsource-variable/noto-sans-thai` |
| `bengali` | `"Noto Sans Bengali Variable"` | `@fontsource-variable/noto-sans-bengali` |
| `cyrillic` | `"Inter Variable"` | `@fontsource-variable/inter` |
| `greek` | `"Inter Variable"` | `@fontsource-variable/inter` |
| `hebrew` | `"Noto Sans Hebrew Variable"` | `@fontsource-variable/noto-sans-hebrew` |

`setupFontStackCSS()` reads the writing system's `defaultFont` and sets `--font-primary` on `<html>`.

### Adding a new writing system

Example: adding Bengali (`beng`).

1. **Add the package to `package.json`:**
   ```json
   "@fontsource-variable/noto-sans-bengali": "^5.2.10"
   ```

2. **Add the entry to `WRITING_SYSTEMS`** in `src/scripts/lib/i18n/writing-systems.ts`:
   ```ts
   {
     code: 'bengali',
     name: 'Bengali Script',
     nameId: 'Skrip Bengali',
     description: 'Bengali script used in Bengali and Assamese languages',
     languages: [],
     numberingSystems: ['beng'],
     direction: DIRECTION_CODE.LTR,
     defaultFont: 'Noto Sans Bengali Variable',
   }
   ```

3. **Add the entry to `NOTO_SANS`** in `src/scripts/lib/i18n/fonts.ts` (remember: both `css` and `loader` need the literal string):
   ```ts
   beng: {
     css: '@fontsource-variable/noto-sans-bengali/index.css',
     loader: () => import('@fontsource-variable/noto-sans-bengali/index.css'),
   },
   ```
   The `SCRIPT_FONT_LOADERS` and `SCRIPT_FONT_CSS` maps are derived from this automatically.

4. **Add at least one locale** with `nativeNumberingSystem: NUMBERING_SYSTEM_CODE.BENG` in `src/scripts/lib/i18n/data.ts`.

5. **Run `bun run build`** to verify the font bundle is created in `dist/assets/fonts/`.
