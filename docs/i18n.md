# i18n

This document describes the internationalization system used in this project. It covers the source locale files, Nunjucks template helpers, runtime language switching, locale-aware formatting, regional pricing, generated types, and maintenance workflow.

The system is designed around one source of truth: locale content lives in `src/locales`, locale behavior lives in `src/configs/locales.ts`, and the same translation keys support both server-rendered HTML and client-side language switching.

## Overview

| Area | File | Purpose |
| --- | --- | --- |
| Locale configuration | `src/configs/locales.ts` | Locale codes, currencies, timezones, directions, numbering systems, plural rules, fallbacks |
| Source translations | `src/locales/{locale}` | JSON5 files edited by developers |
| Template helpers | `src/scripts/lib/template.ts` | Nunjucks `i18n.*` API and runtime data attributes |
| Runtime i18n | `src/scripts/lib/i18n.ts` | i18next initialization, DOM translation, formatter refresh |
| Alpine store | `src/scripts/stores/i18n.ts` | `$store.i18n.languages`, `$store.i18n.current`, `$store.i18n.change()` |
| Formatting utilities | `src/scripts/utils/i18n-format.ts` | Intl formatting, native digits, regional prices, pluralize helpers |
| Locale utilities | `src/scripts/utils/locale.ts` | Locale fallback, currency, timezone, direction, plural suffix helpers |
| Generated types | `generated/i18n.d.ts` | TypeScript translation key definitions |
| Exchange rates | `generated/exchange-rates.ts` | Currency conversion data |

The default locale is `id-ID`. The configured locale list currently contains 43 BCP 47 locale codes, including `id-ID`, `en-US`, `zh-Hans-CN`, `zh-Hant-TW`, `ar-SA`, and `ja-JP`.

## Runtime Model

1. Developers write source copy in `src/locales/{locale}/*.json5`.
2. Nunjucks templates call helpers such as `{{ i18n.t('page.hero.title') }}`.
3. The build renders initial HTML using the default locale.
4. Template helpers emit data attributes such as `data-i18n` and `data-format-number`.
5. The inline i18n script exposes `window.__I18N_DATA__`, `window.__PAGE_ID__`, and `window.__SAVED_LOCALE__`.
6. Runtime i18next loads the inline resources and updates translated elements.
7. The Alpine store switches languages without a full page reload.

Example:

```njk
{{ i18n.t('page.hero.title') }}
```

Output:

```html
<span data-i18n="home:hero.title">Build multilingual pages faster</span>
```

When the user changes language, the runtime reads `data-i18n="home:hero.title"`, resolves the new translation through i18next, and updates the element in place.

## Locale Files

Use this structure:

```txt
src/locales/{locale}/
  common.json5
  home.json5
  about.json5
  pricing.json5
  components/
    cta.json5
```

Key resolution:

| Template key | Runtime namespace | Source file |
| --- | --- | --- |
| `site_name` | `common:site_name` | `src/locales/{locale}/common.json5` |
| `page.hero.title` | `{page}:hero.title` | `src/locales/{locale}/{page}.json5` |
| `comp.cta.title` | `components/cta:title` | `src/locales/{locale}/components/cta.json5` |

Example:

```json5
// src/locales/id-ID/home.json5
{
  hero: {
    title: "Build multilingual pages faster",
    subtitle: "One translation source for server rendering and runtime switching.",
  },
  stats: {
    projects_one: "{{count}} project",
    projects_other: "{{count}} projects",
  },
}
```

## Quick Start

```njk
<h1>{{ i18n.t('page.hero.title') }}</h1>
<p>{{ i18n.t('page.hero.subtitle') }}</p>

<img src="/hero.webp" {{ i18n.attr('page.hero.image_alt', 'alt') }}>

<strong>{{ i18n.formatNumber(1234567) }}</strong>
<span>{{ i18n.formatCurrency(500000, 'IDR') }}</span>
<time>{{ i18n.formatDateTime('2026-06-04T09:00:00Z') }}</time>

<button type="button" @click="$store.i18n.change('en-US')">
  English
</button>
```

Use `i18n.t()` for visible page copy. Use `i18n.text()` or `i18n.attr()` for attributes and non-wrapped strings. Use formatter helpers for numbers, currencies, dates, units, lists, durations, and prices.

## Translation API

| API | Example | Result |
| --- | --- | --- |
| `i18n.t(key, vars?, options?)` | `{{ i18n.t('page.title') }}` | Wrapped HTML with `data-i18n` |
| `i18n.text(key, vars?)` | `{{ i18n.text('page.title') }}` | Plain string |
| `i18n.html(key, vars?)` | `{{ i18n.html('page.body') }}` | Raw locale string |
| `i18n.attr(key, attrName, vars?)` | `<img {{ i18n.attr('page.alt', 'alt') }}>` | `alt="..."` |
| `i18n.plural(key, count, vars?, options?)` | `{{ i18n.plural('page.stats.projects', count) }}` | Plural text with `data-i18n-plural` |
| `i18n.getPluralSuffix(count)` | `{{ i18n.getPluralSuffix(2) }}` | Suffix such as `_other`, `_one`, `_few` |

Options for `i18n.t()`:

| Option | Description |
| --- | --- |
| `raw: true` | Return a string without the wrapper span |
| `native: true` | Force native digits |
| `universal: true` | Force Latin digits (`0-9`) |
| `className: '...'` | Add a class to the wrapper span |

Variables:

```json5
{
  welcome: "Hello {{name}}",
}
```

```njk
{{ i18n.t('welcome', { name: 'Dinda' }) }}
```

Plural keys:

```json5
{
  items_one: "{{count}} item",
  items_other: "{{count}} items",
}
```

```njk
{{ i18n.plural('page.items', total) }}
```

Locales with richer plural rules should provide the required suffixes, such as `_zero`, `_one`, `_two`, `_few`, `_many`, and `_other`.

## Formatting API

The following helpers are available in Nunjucks templates. Most formatter helpers render a `<span>` with runtime data attributes so the value can be reformatted when the active locale changes. Pass `{ raw: true }` when a plain string is required.

| API | Example | Use case |
| --- | --- | --- |
| `formatNumber(value, options?)` | `{{ i18n.formatNumber(12000) }}` | General numbers |
| `formatCurrency(value, currency, options?)` | `{{ i18n.formatCurrency(29, 'USD') }}` | Currency values |
| `formatPercent(value, options?)` | `{{ i18n.formatPercent(0.85) }}` | Percentages (`0.85` becomes `85%`) |
| `formatBytes(bytes, decimals?, options?)` | `{{ i18n.formatBytes(2048) }}` | File sizes |
| `formatDuration(seconds, options?)` | `{{ i18n.formatDuration(3661) }}` | Durations |
| `formatDate(date, options?)` | `{{ i18n.formatDate(page.published_at) }}` | Dates |
| `formatTime(date, preset?, options?)` | `{{ i18n.formatTime(date, 'short') }}` | Times with `short`, `medium`, `long`, or `full` presets |
| `formatDateTime(date, options?)` | `{{ i18n.formatDateTime(date) }}` | Date and time values |
| `formatOrdinal(value, options?)` | `{{ i18n.formatOrdinal(1) }}` | Rankings and ordered labels |
| `formatCardinal(value, options?)` | `{{ i18n.formatCardinal(12) }}` | Number words for supported locales |
| `formatScientific(value, options?)` | `{{ i18n.formatScientific(1200000) }}` | Scientific notation |
| `formatAbbreviated(value, options?)` | `{{ i18n.formatAbbreviated(15300) }}` | Compact numbers such as `15.3K` |
| `formatList(items, options?)` | `{{ i18n.formatList(['A', 'B', 'C']) }}` | Locale-aware lists |
| `formatUnit(value, unit, options?)` | `{{ i18n.formatUnit(25, 'celsius') }}` | Intl units such as `kilometer`, `celsius`, `byte` |
| `formatRelativeTime(value, options)` | `{{ i18n.formatRelativeTime(-1, { unit: 'day', numeric: 'auto' }) }}` | Relative time |
| `nativeDigits(text, force?, options?)` | `{{ i18n.nativeDigits('Page 10', true) }}` | Initial native digit conversion |
| `pluralize(word, count?, inclusive?, options?)` | `{{ i18n.pluralize('project', 2) }}` | Initial English pluralization |
| `singularize(word, options?)` | `{{ i18n.singularize('projects') }}` | Initial English singularization |

Common formatter options:

| Option | Description |
| --- | --- |
| `raw: true` | Return a string without a wrapper span |
| `className: '...'` | Add a class to the wrapper span |
| `nativeDigits: true` | Use the locale native numbering system when available |
| `numberingSystem: 'latn'` | Override the Intl numbering system |

Intl options can be passed to date and number formatters:

```njk
{{ i18n.formatDate('2026-06-04', { dateStyle: 'long' }) }}
{{ i18n.formatNumber(1200, { maximumFractionDigits: 0 }) }}
```

Runtime note: `nativeDigits()`, `pluralize()`, and `singularize()` render initial output only. They are not refreshed by `updateFormattedElements()`. Use translation keys with `i18n.t()` or `i18n.plural()` for content that must change during language switching.

## Pricing API

Use `pricing.base` as the USD fallback. Add locale-specific prices when needed.

```json5
{
  pricing: {
    base: 29,
    "id-ID": 500000,
    "ja-JP": 4500,
  },
}
```

| API | Example | Use case |
| --- | --- | --- |
| `localPrice(plan, options?)` | `{{ i18n.localPrice(plan) }}` | Raw local price |
| `localPriceCurrency(plan, options?)` | `{{ i18n.localPriceCurrency(plan) }}` | Currency for the selected local price |
| `formatLocalPrice(plan, options?)` | `{{ i18n.formatLocalPrice(plan) }}` | Local price formatted with currency |
| `convertCurrency(value, targetCurrency?, options?)` | `{{ i18n.convertCurrency(29, 'IDR') }}` | Convert from base currency to target currency |
| `convertLocalPrice(plan, targetCurrency?, options?)` | `{{ i18n.convertLocalPrice(plan, 'USD') }}` | Convert local price to target currency |
| `formatLocalPriceDiscounted(plan, multiplier, targetCurrency?, options?)` | `{{ i18n.formatLocalPriceDiscounted(plan, 0.8, 'IDR') }}` | Discounted price |

Exchange rates are read from `generated/exchange-rates.ts`.

```bash
bun run fetch:rates
bun run fetch:rates:force
```

## Runtime Switcher

The Alpine i18n store is available after app startup.

```njk
<div x-data>
  <template x-for="lang in $store.i18n.languages" :key="lang.code">
    <button
      type="button"
      :aria-pressed="$store.i18n.current === lang.code"
      @click="$store.i18n.change(lang.code)"
    >
      <span x-text="lang.label"></span>
    </button>
  </template>
</div>
```

Store API:

| Property | Type | Description |
| --- | --- | --- |
| `$store.i18n.languages` | `Array<{ code, label, flag }>` | Locales from `LOCALES` |
| `$store.i18n.current` | `string` | Locale from `localStorage` or fallback |
| `$store.i18n.change(code)` | `function` | Save locale, update i18next, update HTML attributes, refresh translations and formatters |

When `change(code)` runs, the store:

1. Stores the locale in `localStorage` using `i18nextLocale`.
2. Calls `i18next.changeLanguage(code)`.
3. Updates `<html lang>`, `<html dir>`, `data-script`, and the `is-rtl` class.
4. Runs `translatePage()`.
5. Runs `updateFormattedElements()`.
6. Refreshes language labels in the store.

## Locale Configuration

Locales follow BCP 47 format: `language-script-region` (e.g., `zh-Hans-CN`). The `script` field is required for Chinese locales (`Hans` for Simplified, `Hant` for Traditional) and optional for others.

Add or update locales in `src/configs/locales.ts`. A locale entry defines:

```ts
{
  code: 'id-ID',
  language: LANGUAGE_CODE.ID,
  region: REGION_CODE.ID,
  flag: FLAG_CODE.ID,
  dir: DIRECTION_CODE.LTR,
  currency: CURRENCY_CODE.IDR,
  numberingSystem: NUMBERING_SYSTEM_CODE.LATN,
  nativeNumberingSystem: NUMBERING_SYSTEM_CODE.LATN,
  nativeDigits: false,
  timezone: TIMEZONE_CODE.ASIA_JAKARTA,
  timezoneOffset: 7,
  calendar: CALENDAR_CODE.GREGORY,
  dateFormat: 'short',
  timeFormat: 'short',
  firstDayOfWeek: 1,
  pluralRules: PLURAL_RULE_CODE.OTHER,
}
```

For Chinese locales with script variants, include the `script` field:

```ts
{
  code: 'zh-Hans-CN',
  language: LANGUAGE_CODE.ZH,
  script: SCRIPT_CODE.HANS,  // Required for Chinese script variants
  region: REGION_CODE.CN,
  // ... other fields
}
```

Important constants:

| Constant | Current value or source |
| --- | --- |
| `DEFAULT_LOCALE` | `id-ID` |
| `BASE_CURRENCY` | `USD` |
| `LOCALE_STORAGE_KEY` | `i18nextLocale` |
| `LOCALE_CODES` | Derived from `LOCALES` |
| `LOCALE_FALLBACKS` | Explicit fallback mapping for unsupported regional variants |

`getLocale()` always returns a configured locale. Unsupported locale input falls back through `LOCALE_FALLBACKS`, then by language subtag, then to `DEFAULT_LOCALE`.

## RTL and Font Behavior

The runtime updates the root element during language switching:

```html
<html lang="ar-SA" dir="rtl" data-script="latin" class="is-rtl">
```

Implementation note: `data-script` is currently derived from `numberingSystem`, not `nativeNumberingSystem`. Because many locales use `numberingSystem: LATN`, `data-script` can remain `latin` even when a locale has native digits. Keep `src/configs/locales.ts` and `src/styles/main.css` aligned if automatic script-specific font selection is required.

Use logical CSS properties for layout that must work in both LTR and RTL:

```css
.card {
  padding-inline: 1rem;
  margin-inline-start: 2rem;
  border-inline-end: 1px solid currentColor;
}
```

Avoid physical directional properties for multilingual layout:

```css
.card {
  margin-left: 2rem;
  border-right: 1px solid currentColor;
}
```

## Type Safety

`bun run gen:i18n` reads the default locale directory, `src/locales/id-ID`, and generates `generated/i18n.d.ts`.

The generated file contains:

- `I18nTranslationKeys`
- `I18nCommon`
- `I18nPages`
- `I18nComponents`

Client-side `t()` in `src/scripts/lib/i18n.ts` uses `I18nTranslationKeys`, so TypeScript code can reference generated translation keys.

## Workflow

Add page copy:

1. Edit `src/locales/id-ID/{page}.json5`.
2. Add the same key structure to other locale files.
3. Use `{{ i18n.t('page.section.key') }}` in the template.
4. Run `bun run gen:i18n`.
5. Run `bun run check:parity`.

Add reusable component copy:

1. Create or edit `src/locales/id-ID/components/{name}.json5`.
2. Make sure the component is included by the page template.
3. Use `{{ i18n.t('comp.{name}.key') }}`.
4. Run `bun run gen:i18n`.

Add a new locale:

1. Add the locale entry in `src/configs/locales.ts`.
2. Run `bun run sync:locales` to copy the default locale structure.
3. Translate the new JSON5 files.
4. Run `bun run gen:i18n`.
5. Run `bun run check:parity`.

## Best Practices

Use i18n for user-facing copy:

```njk
{{ i18n.t('page.hero.title') }}
```

Use formatters instead of manual formatting:

```njk
{{ i18n.formatCurrency(plan.price, 'IDR') }}
{{ i18n.formatDate(page.published_at, { dateStyle: 'medium' }) }}
```

Use `raw` only when wrapper HTML is not allowed:

```njk
<meta name="description" content="{{ i18n.t('page.seo.description', {}, { raw: true }) }}">
```

Use `attr()` for simple attributes:

```njk
<input {{ i18n.attr('page.form.email_placeholder', 'placeholder') }}>
```

Avoid hardcoded user-facing copy:

```njk
<!-- Avoid -->
<h1>Welcome</h1>
```

Avoid manual currency formatting:

```njk
<!-- Avoid -->
Rp {{ price }}
```

## Troubleshooting

| Issue | Common cause | Fix |
| --- | --- | --- |
| `[home:hero.title]` is displayed | Missing key in the active runtime resource | Add the key to the relevant locale and run `bun run gen:i18n` |
| SSR text is correct but does not change after switching language | The output has no runtime data attribute | Use `i18n.t()` or a formatter helper |
| Attribute text does not change after switching language | `i18n.attr()` only renders the initial attribute | Use the manual `data-i18n-attr` runtime pattern when the attribute must update |
| Number format does not change | The number is manually rendered | Use `i18n.formatNumber()` or another formatter helper |
| Price uses the wrong currency | The plan has no locale price and falls back to `base` | Add a locale-specific price or use a conversion helper |
| RTL layout is broken | CSS uses physical direction properties | Use logical properties such as `margin-inline-start` |
| Build fails | Invalid JSON5 or stale generated types | Fix JSON5 and run `bun run gen:i18n` |
| Parity check fails | Locale files have missing or extra keys | Run `bun run check:parity -- --verbose` and align the key structure |

## Review Checklist

Before merging i18n changes:

- User-facing copy uses `i18n.*`.
- `src/locales/id-ID` contains the complete default key structure.
- Other locales match the same key structure.
- Formatters are used for numbers, currencies, dates, units, lists, durations, and prices.
- New layout CSS uses logical properties where RTL can apply.
- `bun run gen:i18n` has been run.
- `bun run check:parity` passes, or known parity issues are documented.
